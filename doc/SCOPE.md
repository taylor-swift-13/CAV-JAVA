# CAV Scope

本文件定义：这个工具链适合处理什么样的 C 函数，以及 Contract / Verify 两阶段的职责边界。

## 1. 这个工具适合什么程序

当前 CAV 工作流最适合：

- 单个目标函数，语义明确，输入输出边界清楚
- 命令式 C，控制流以顺序、分支、`for` / `while` 为主
- 常见数据形态：标量、数组、字符串、指针、以及仓库已有谓词支持的结构体 / 链表 / 树
- 可以写成输入约束、输出关系、内存拥有关系和少量数学定义的题目

最适合的题型：

- 纯计算函数
- 数组扫描 / 前缀处理 / 双重循环
- 字符串扫描 / 构造
- 读输入数组、写输出数组
- 在调用者提供的缓冲区上做原地更新
- 需要少量题目专用数学定义的算法题

## 2. 不优先支持的写法

下面这些写法会显著提高 contract 和 verify 难度，应尽量避免：

- I/O、测试驱动、命令行主程序、平台依赖逻辑
- 大量宏、复杂条件编译、函数指针回调
- 依赖外部全局状态或隐藏副作用
- 频繁申请和释放内存
- 返回字符串字面量、返回静态内部缓冲区这类只读静态存储接口
- 把资源创建、资源填充、资源释放混在同一个待验证函数里

如果可以改写，应先改写成验证友好的接口。

## 3. 实现也应尽量写成验证稳定的等价形式

如果两种实现写法语义等价，应优先选择 verify 前端和 `symexec` 更稳定的那一种。

常见规则：

- 字符串扫描类循环，优先写成
  - `while (1) { if (s[i] == 0) break; ... }`
  - 不要优先写成 `while (s[i] != '\0')`
- 字符终止符优先直接写 `0`
  - 不要依赖 `'\0'`
- 如果字符常量会让前端处理不稳定，可以直接写对应整数值
  - 例如 `'a'` 可以写成 `97`

这类改写只改变实现表面形式，不改变程序语义；目的只是减少 annotation、symexec 和 proof 的前端噪声。

## 4. 算法题的内存使用方式

算法题里的内存通常有两种写法：

1. 作为输入参数传入
2. 在函数中通过已规格化的 `malloc_*` 包装函数申请

两种方式都允许，选择时只要和题目语义一致即可。

常见作为输入参数传入的内存包括：

- 输出数组
- 临时缓冲区
- 工作区
- 辅助数组
- `dp` 数组
- 字符串输出缓冲区

例如：

- `void hello(char *out)`
- `int climb(int n, int *dp)`

## 5. 使用 malloc 的规则

如果选择在函数中申请内存，应按包装函数使用。

规则：

- 不要直接裸写系统 `malloc`
- 先声明一个带规格的 `malloc_*` 包装函数
- 再在目标函数里调用它

最小模板例如：

```c
int *malloc_int_array(int n)
/*@ Require n > 0 && emp
    Ensure exists l, IntArray::full(__return, n, l)
*/
;
```

或：

```c
char *malloc_char_array(int n)
/*@ Require n > 0 && emp
    Ensure exists l, CharArray::full(__return, n, l)
*/
;
```

如果题目语义是“返回新分配对象”，就可以用这种方式。

## 6. 两段流程

本目录采用严格的两段流程：

- Contract：生成验证输入
- Verify：验证规格输入

## 7. Contract 负责什么

Contract 负责：

- 从原始题意、原始代码和自然语言描述出发
- 判断当前接口是否适合验证
- 必要时把原始代码改写成验证友好的接口形态
- 生成 `input/<name>.c`
- 必要时生成 `input/<name>.v`
- 写完整前后条件

如果原始接口不适合验证，Contract 应优先做的改写包括：

- 去掉 I/O
- 去掉测试脚手架
- 去掉平台依赖逻辑
- 优先把需要的内存改成输入参数
- 必要时把“返回静态字面量/内部缓冲区”改成调用者提供输出缓冲区或显式分配对象

## 8. Verify 负责什么

Verify 负责：

- 消费 `input/<name>.c` 和可选 `input/<name>.v`
- 补 `Assert` / `Inv`
- 运行 symbolic / qcp
- 完成 manual proof

## 9. 核心边界

- 重新设计接口与规格属于 Contract
- 验证与证明属于 Verify
- 不要在 Verify 中回退成“边修规格边验证”的混合流程
